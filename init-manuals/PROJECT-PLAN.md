# 프로젝트 계획서

## 1. 프로젝트 개요

### 1.1 프로젝트 명
합주 일정 캘린더 (Band Practice Schedule Calendar)

### 1.2 프로젝트 목적
밴드부 내 여러 팀의 합주 일정을 시각적으로 관리하여 가독성과 편의성을 향상시키는 웹 애플리케이션 개발

### 1.3 개발 배경
- 현재 단체 채팅방에서 텍스트로만 일정을 관리하고 있어 가독성이 떨어짐
- 여러 팀의 일정이 중복되거나 충돌하는 경우 확인이 어려움
- 한눈에 일정을 파악할 수 있는 시각적 도구의 필요성

### 1.4 프로젝트 범위
- 팀 이름, 날짜, 시간을 입력받아 달력에 표시하는 기능
- 달력 형태의 시각적 인터페이스 제공
- 일정 조회 및 상세 정보 표시 기능
- 특정 날짜 클릭 시 해당 날짜의 일정을 시간대별로 확대하여 보는 기능

## 2. 프로젝트 일정

### 2.1 개발 단계
| 단계 | 활동 | 예상 기간 |
|------|------|-----------|
| 1단계 | 요구사항 분석 및 설계 | 1주 |
| 2단계 | 백엔드 개발 (Spring Boot) | 2주 |
| 3단계 | 프론트엔드 개발 (Thymeleaf) | 2주 |
| 4단계 | 통합 및 테스트 | 1주 |
| 5단계 | 배포 및 문서화 | 1주 |

### 2.2 주요 마일스톤
- 요구사항 분석 완료
- 데이터베이스 설계 완료
- 백엔드 API 개발 완료
- UI/UX 구현 완료
- 최종 테스트 및 배포

## 3. 개발 환경

### 3.1 기술 스택
- **백엔드**: Spring Boot 3.x, Java 17+
- **프론트엔드**: Thymeleaf, HTML5, CSS3, JavaScript
- **빌드 도구**: Gradle
- **데이터베이스**: H2 Database (개발), MySQL/PostgreSQL (운영)
- **버전 관리**: Git

### 3.2 개발 도구
- IDE: IntelliJ IDEA / VS Code
- API 테스트: Postman
- 형상 관리: GitHub

## 4. 프로젝트 조직

### 4.1 역할 및 책임
- **개발자**: 백엔드/프론트엔드 개발, 테스트
- **사용자**: 밴드부 팀장 및 부원들

### 4.2 의사소통 계획
- 주간 진행 상황 리뷰
- 사용자 피드백 수집 및 반영

## 5. 위험 관리

### 5.1 주요 위험 요소
| 위험 | 영향도 | 대응 방안 |
|------|--------|-----------|
| 요구사항 변경 | 중 | 애자일 방식으로 유연하게 대응 |
| 기술적 어려움 | 중 | 충분한 학습 시간 확보 및 문서 참고 |
| 일정 지연 | 중 | 우선순위 기반 개발, MVP 먼저 구현 |

### 5.2 품질 관리
- 코드 리뷰 실시
- 단위 테스트 작성
- 사용자 테스트를 통한 피드백 수집

## 6. 성공 기준

### 6.1 기능적 요구사항 달성
- 일정 등록, 조회, 수정, 삭제 기능 구현
- 달력 형태의 직관적인 UI 제공
- 일정 충돌 확인 가능

### 6.2 비기능적 요구사항 달성
- 응답 시간 1초 이내
- 직관적이고 사용하기 쉬운 인터페이스
- 모바일 브라우저에서도 사용 가능한 반응형 디자인

## 7. 배포 및 운영

### 7.1 배포 전략
- 로컬 환경에서 충분한 테스트 후 배포
- 단계적 배포 (알파 → 베타 → 정식)

### 7.2 유지보수 계획
- 사용자 피드백 기반 개선
- 정기적인 버그 수정 및 기능 개선
